NAME - ::<namespace>:: L47_client_mapping

DESCRIPTION

IxLoad

This command is used to construct the mapping between the agents that will generate the traffic and the set of networks which will carry the traffic. A number of additional options control the manner in which the traffic is applied to the networks. The command can specify the options used to allow the test to achieve a particular objective, for example, connections per second, or the options that determine the timeline for application of traffic or the options that control the manner in which client traffic is sent to servers.

-objective_type

Supported platforms Details

IxLoad

DESCRIPTION

The objective to be achieved for this traffic to network mapping. Some objectives may not be available for some trafficnetwork pairings. (See Notes from the beginning of procedure description)Dependencies: -mode add | modifyNA - if this option is not used there will be no objective set.users - The objective is to simulate some number of users. If you select this objective, remember that a `user' does not necessarily mean one human user. For example, a web browser being used by one person may open several connections to a web site simultaneously; each connection counts as one `user', because each connection was initiated by the same source simultaneously. Specify the desired number of users in the objective_value option.connections - The objective is to sustain some number of connections simultaneously. Specify the desired number of connections in the objective_value option.sessions - The objective is to sustain some number of sessions simultaneously. Specify the desired number of connections in the objective_value option.crate - The objective is to create connections at a certain rate. For example, if the traffic in this mapping is FTP client traffic, this objective will attempt to generate the specified number of FTP connections per second. Specify the desired number of connections per second in the objective_value option.tputmb - The objective is to achieve a certain level of throughput, measured in Megabytes per second (MBps). Specify the amount of throughput in the objectiveValue option.tputkb - The objective is to achieve a certain level of throughput, measured in Kilobytes per second (KBps). Specify the amount of throughput in the objectiveValue option.trate - The objective is to complete transactions at a certain rate. For example, if the traffic in this mapping is FTP client traffic, this objective will attempt to complete the specified number of transactions per second.

DEFAULT

 None

In this case, the -agent_handle parameter must be a list containing a number of handles equal to this parameter, all agents in the list being part of the same client.Some objectives may not be available for some trafficnetwork pairings. (See Notes from the beginning of procedure description)Dependencies: -mode add | modify; -agent_handle "an agent handle"; handle "a client traffic handle"

-port_map_policy

Supported platforms Details

IxLoad

DESCRIPTION

This option controls the sequence in which the client ports connect to the server ports.pairs = Client agents connect to server agents on a one-to-one basis.mesh = Agents on every client port connect to every server port.For large numbers of ports, the Port Pair sequence scales performance better than the Port Mesh sequence.The operation of Port Pairs can be described by three scenarios:If the number of client ports is equal to the number of server agents, client ports will establish connections to server ports on a one-to-one basis.If the number of client ports is less than the number of server ports, the client ports will establish connections to the server ports on a one-to-one basis until all client ports are paired with server ports. The remaining server ports will not be used.If the number of client ports is greater than the number of server ports, the client ports will establish connections to the server ports on a one-to-one basis until all server ports are paired with client ports. Then, the remaining client ports will return to the first server port and continue pairing themselves with server portsDependencies: -mode add | modify

DEFAULT

pairs)

-ramp_down_time

Supported platforms Details

IxLoad

DESCRIPTION

The amount of time used for closing any TCP connections that are still open after all transactions are complete. When the ramp down time expires, the program terminates any users. If the program terminates any client users that are still running after the ramp down time expires , statistics for servers and clients that should match may not. This is an indication that the ramp down time may be too short.Dependencies: -mode add | modify

DEFAULT

20)

-ramp_up_interval

Supported platforms Details

IxLoad

DESCRIPTION

This field accepts integer values. The value for this option will be considered only when -ramp_up_type is users_per_second. You can edit the value to increment or decrement the number of users to be started at every ramp_up_interval. Dependencies: -mode add | modify

DEFAULT

 None

-ramp_up_type

Supported platforms Details

IxLoad

DESCRIPTION

The method used to apply the rampUpValue.users_per_second = applies the rampUpValue to bring up the specified number of users per second. For example, if you select users_per_second and you specify 10 for the ramp_up_value, the program brings up 10 new users every second until all the users are up and running.Dependencies: -mode add | modify

DEFAULT

 None

-ramp_up_value

Supported platforms Details

IxLoad

DESCRIPTION

Dependencies: -mode add | modify

DEFAULT

 None

-standby_time

Supported platforms Details

IxLoad

DESCRIPTION

The amount of time, expressed in seconds, that elapses between the time the test is started and the time that the traffic-network pair become active. If you have multiple traffic-network paris in your test, you can use this parameter to stagger their start times. A value of 0 causes the test to begin immediately. The valid range is from 0 to 1000 hours (3,600,000). Default: 0Dependencies: -mode add | modify

DEFAULT

0)

-sustain_time

Supported platforms Details

IxLoad

DESCRIPTION

The amount of time, in seconds, during which all users are up and performing the central test objectives, such as establishing and closing connections (TCP), retrieving or serving pages (HTTP), or sending or receiving files (FTP). The valid range is from 0 to 1000 hours (3,600,000).Dependencies: -mode add | modify

DEFAULT

0)

port_map_policy

Supported platforms Details

IxLoad

DESCRIPTION

This option controls the sequence in which the client ports connect to the server ports.pairs = Client agents connect to server agents on a one-to-one basis.mesh = Agents on every client port connect to every server port.For large numbers of ports, the Port Pair sequence scales performance better than the Port Mesh sequence.The operation of Port Pairs can be described by three scenarios:If the number of client ports is equal to the number of server agents, client ports will establish connections to server ports on a one-to-one basis.If the number of client ports is less than the number of server ports, the client ports will establish connections to the server ports on a one-to-one basis until all client ports are paired with server ports. The remaining server ports will not be used.If the number of client ports is greater than the number of server ports, the client ports will establish connections to the server ports on a one-to-one basis until all server ports are paired with client ports. Then, the remaining client ports will return to the first server port and continue pairing themselves with server portsDependencies: -mode add | modify
