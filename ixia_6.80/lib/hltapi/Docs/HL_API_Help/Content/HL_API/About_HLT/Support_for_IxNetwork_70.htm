Support for IxNetwork 7.0

See IxAccess to IxNetwork Migration, and Sample Scripts, for details and examples.

IxNetwork 7.0 contains an entirely new API in addition to the old API. From a Tcl package perspective, the old one is IxTclProtocol and the new one is IxTclNetwork (see Old API vs. New API). After executing the new api, you will be able to view and interact with the configuration in IxNetwork, just as you had been able to do in the past with IxExplorer.

HLT 3.0 was the first of several planned releases that rolled out support for the entire new API. In HLT 3.0, the BGP, IS-IS, LDP, OSPF, RIP and RSVP protocols were exposed. In HLT 3.10, BFD, and IGMP were exposed. In HLT 3.20, LACP, EIGRP, PIM, P2MP-TE, RIPng and STP were exposed. In HLT 3.30, L2TP and PPP were exposed. In HLT 3.40, MLD, OSPFv3, CFM, OAM, ANCP, MVPN, DHCP (v4 and v6), DCE-ISIS, and TWAMP support were exposed.

The IxTclNetwork API does not intermix with the other Tcl APIs.

HLT supports both the Tcl APIs that can be run through IxNetwork. To determine which one you use, the HLTSET values for IxNetwork have been enhanced. You now see a P (IxTclProtocol, old API path) or a N (IxTclNetwork, new API path) after the IxNetwork version in the HLTSET.

A new HLTSET type has been added, this is the similar to the N version and is called NO (IxTclNetwork API only) . The only difference is that now you can only create traffic by using-traffic_generator ixnetwork or ixnetwork_540. The-traffic_generator ixos option is not supported anymore with this HLTSET version. This HLTSET improves performance for chassis connection and port assignment on multiple port setups.

The old IxNetwork package, IxTclProtocols, is located inside the IxOS installation lib folder. However, the new API package, IxTclNetwork, is located inside the IxNetwork installation lib folder.

Old API vs. New API

The following summary describes the differences between the Old API (IxTclProtocol) and the New API (IxTclNetwork).

OLD API

  • Solaris
    • tcl_server-must be present, and if not present, is by default equal to the first item in the device(chassis) list
    • ixnetwork_tcl_server-is ignored
  • Windows
    • tcl_server-is ignored
    • ixnetwork_tcl_server-is ignored

NEW API

  • Solaris
    • tcl_server-must be present, and if not present, is by default equal to ixnetwork_tcl_server
    • ixnetwork_tcl_server-if not present is by default:
      • equal to 127.0.0.1 if tcl_server is not present
        or
      • equal to tcl_server, if tcl_server is present
  • Windows
    • tcl_server
      • is ignored if enable_win_tcl_server is 0 (default)
      • if enable_win_tcl_server is 1, and tcl_server is not present, is by default equal to ixnetwork_tcl_server
    • ixnetwork_tcl_server-if not present is by default:
      • equal to 127.0.0.1 if tcl_server is not present (this is the most frequent case on Windows)
        or
      • equal to tcl_server, if tcl_server is present

Notes for traffic_config

  • When using IxNetwork, traffic configurations can be done by using previously created handles (IP interfaces, PPP ranges, L2TP ranges, Protocol Route Ranges , and so on) as sources (parameter -emulation_src_handle) and destinations (-emulation_dst_handle). The port_handle parameter is not necessary anymore.
  • When using IxNetwork, traffic can only be configured between Ixia end points.
  • The parameter -traffic_generator was introduced in order to choose between old and new APIs. Valid choices are as follows:
    • ixos-old API is used
    • ixnetwork-new API is used
    • In the traffic_config call, ICMP options are valid for IPv4 only. ICMP options are not currently supported for IPv6 streams.

Under each parameter description, you can find the supported traffic_generator options.
